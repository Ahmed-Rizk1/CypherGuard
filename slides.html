A Security Crisis in Real Time

Modern cloud applications handle millions of connection requests every minute. Spotting a single malicious attacker within this sea of packets is like finding a specific lockpick carrying thief in a stadium crowd.

This presentation shows our journey in building a hybrid cyber security gateway combining rapid Machine Learning anomalies filtering with generative AI explanation models.

ACT I: THE CHALLENGE

The Signature Trap

02 / 16

  • Rigid Signature Checks Fail

    Traditional firewalls look for exact strings (signatures). If a hacker changes a single character in their script or launches a novel "Zero-Day" attack, signature databases fail instantly.

  • The "Black Box" Problem of AI

    Simple Machine Learning anomaly detection is fast but acts as a black box. It blocks the user, but security teams cannot tell *why* it blocked them, causing delay in alert resolution.

ACT III: THE GATEKEEPER

Leveling the Playing Field

06 / 16
StandardScaler (Z-Score)

Features have radically different bounds. Small Packet Ratio lies in [0, 1], while Flow Bytes/s ranges in millions. StandardScaler centers each feature around a mean of 0 and standard deviation of 1.

This normalization prevents high-volume features from drowning out critical, small-scale timing features (such as IAT metrics) during calculations.

z = (x - μ) / σ
[ 1500B, 1.2s, 0.94 ] ➔ [ 1.4, -0.8, 2.1 ]

ACT III: THE GATEKEEPER

Live Concept Drift Detection

09 / 16
Kullback-Leibler (KL) Divergence

Compares live feature histograms (batches of 1,000 samples) against training baselines in `feature_baselines.json`. If KL divergence exceeds 0.1, it triggers a model retraining alert.

  • Laplace Smoothing

    Adds 1e-10 to all bins to prevent divide-by-zero or log-of-zero errors in mathematically calculating entropy drift.

KL Divergence: 0.024 (Stable)
ACT III: THE GATEKEEPER

Swapping Weapons in Battle

10 / 16

  • 5-Second File Watcher

    A thread watches `model.joblib`. If the modification time changes, a reload is triggered.

  • Atomic Pointer Swap

    The new model is loaded into a separate RAM space. Once fully verified, the model pointer is swapped in less than 1 millisecond. The old model is deleted by the garbage collector.

ACT IV: THE DETECTIVE

The AI Detective & 3-Tier Strategy

11 / 16

The ML Classifier is fast but mute. The **LLM Detective** translates statistics into JSON threat briefs. We use a 3-tier fallback to keep it cheap and reliable:

Instant & Free Response

We round features into buckets. If an identical profile has occurred before, we fetch the response instantly from Redis, skipping external API expenses.

Groq/OpenRouter API Connection

Queries `gpt-4o-mini` with fallback to `llama-3-8b`. Temperature is set to 0.1 for consistency. Output is validated against a Pydantic schema.

Local Fallback Rule Engine

If the API is offline or timeout occurs, rule-based python heuristics evaluate values locally to assign attack labels instantly.

{
  "attack_type": "DDoS Volumetric Flood",
  "severity": "critical",
  "explanation": "High-rate traffic at 5,200 pkt/s saturated bandwidth.",
  "recommendation": "Apply network-level rate limit."
}
ACT IV: THE DETECTIVE

Smart Profile Caching

12 / 16
Bucketing Features

Continuous parameters rarely match exactly. By rounding values into bins (e.g. rounding packets/sec to the nearest 500, average size to the nearest 200), we group similar attacks under one cache key.

This MD5 hash compression yields high cache hit rates and cuts active API expenditure significantly.

Raw (5211 pps, 1345B) ➔ Round (5000, 1400)
MD5 Hash ➔ cache_key_a1b2
ACT IV: THE DETECTIVE

DDoS Cooldown Lock

13 / 16

  • Preventing Flood Exhaustion

    During a DDoS flood at 5,000 packets per second, triggering an LLM request for every packet alert would exhaust API rate limits and budget within seconds.

  • 60-Second Cooldown

    We lock LLM calls for an IP for 60 seconds (`SET NX EX`). The firewall continues to block the traffic, but we suppress duplicate detective analysis.

ACT V: THE COMMANDER

Human-in-the-Loop & Watchdog

14 / 16
Decoupling Autonomy and Safety

When a threat is flagged, a Redis watchdog key is initialized with a 60-second Time-to-Live (TTL). The analyst receives an alert and holds veto power.

Analysts can block manually (A). If they do not respond within 60 seconds, the Redis expiration watch triggers an auto-block (B), ensuring security.
